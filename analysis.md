# Анализ границ ответственности для сценария "Публикация объявления"

## Транзакционные границы

### 1. Где начинается и заканчивается транзакция?
Транзакция начинается в момент получения запроса `POST /api/announcements` контроллером `AnnouncementService` и заканчивается после фиксации записи в базе данных. Успешная фиксация записи является точкой невозврата для основного доменного события.

### 2. Какие операции должны быть атомарными?
Сохранение самого объявления (метаданных: заголовок, текст, автор) и сохранение информации о прикрепленных файлах (если файлы загружаются напрямую в БД или ссылки на них). Это должно быть выполнено в одной транзакции БД, чтобы избежать появления "осиротевших" объявлений без файлов или наоборот.

### 3. Какие операции могут быть асинхронными?
- **Отправка уведомлений участникам.** Это не влияет на факт публикации объявления. Если уведомления отправлять синхронно, это может занять много времени при большой группе.
- **Загрузка файлов в отдельное хранилище (S3/MinIO).** Это IO-операция, которая может быть выполнена после записи в БД или до нее, но с обработкой ошибок.

### 4. Анализ операций

| Операция | Синхронная/Асинхронная | Откат при ошибке | Retry-стратегия | Идемпотентность |
| :--- | :--- | :--- | :--- | :--- |
| **Валидация данных** | Синхронная | Да (остановка потока) | Нет | Нет (зависит от входных данных) |
| **Сохранение метаданных объявления в БД** | Синхронная | Да (ROLLBACK) | Нет (ошибка БД фатальна) | Да (уникальный ID запроса или проверка дублей по хэшу заголовка+текста за короткий промежуток) |
| **Загрузка файла во внешнее хранилище** | Синхронная (для пользователя) / Асинхронная (в архитектуре) | Да (удаление записи из БД, если файл не загрузился) | 3 попытки с экспоненциальной задержкой (1s, 2s, 4s). При провале — отмена всей операции. | Да (генерировать уникальный UUID для имени файла на клиенте) |
| **Отправка уведомлений** | Асинхронная | Нет (best-effort) | 5 попыток с эксп. задержкой через очередь Dead Letter Queue (DLQ) после 3х неудач | Да (дедупликация по `announcement_id` + `user_id` в очереди) |

## Обработка исключительных ситуаций

### Исключительная ситуация: Таймаут при загрузке файла во внешнее хранилище

**Условие**: Внешнее хранилище (S3) не отвечает в течение 5 секунд.

**Обнаружение**:
- HTTP-клиент (например, `RestTemplate` или S3 SDK) выбрасывает `TimeoutException` или `SocketTimeoutException`.
- Логируется ошибка: "File upload timeout for announcement: {temp_id}".

**Реакция**:
1. Система НЕ сохраняет объявление в БД.
2. Система прерывает текущую транзакцию.
3. Система возвращает HTTP 503 (Service Unavailable).

**Компенсация**:
- Так как транзакция в БД не была зафиксирована, никаких данных удалять не нужно.
- Если файл был частично загружен и хранилище поддерживает версионность, система может запустить фоновую задачу на "очистку мусора" для удаления не привязанных к объявлениям файлов, но это обычно делается отдельным скриптом.

**Уведомление пользователя**:
- Администратор видит сообщение: "Не удалось загрузить файл. Сервис временно недоступен. Пожалуйста, попробуйте позже."

### Исключительная ситуация: Ошибка подключения к очереди сообщений (RabbitMQ)

**Условие**: После сохранения объявления в БД, система не может опубликовать задачу в очередь `notifications`.

**Обнаружение**:
- При вызове `rabbitTemplate.convertAndSend()` выбрасывается `AmqpException`.

**Реакция**:
1. Система логирует критическую ошибку.
2. Система НЕ откатывает транзакцию БД (объявление уже опубликовано).
3. Система помещает задачу на отправку уведомлений в локальную базу данных (таблица `outbox`) со статусом `PENDING`.

**Компенсация**:
- Запускается фоновый процесс (Scheduler), который раз в минуту сканирует таблицу `outbox` и пытается повторно отправить неотправленные сообщения в RabbitMQ.
- После успешной отправки статус в `outbox` меняется на `SENT`.

**Уведомление пользователя**:
- Пользователь видит сообщение об успешной публикации, так как доменная операция выполнена.
- Системный администратор получает алерт о падении очереди.
