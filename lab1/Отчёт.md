### 2. Use-case описание

👉 **Ссылка на файл:** [use-case.md](use-case.md)

**Основной сценарий:** Публикация объявления в группе

**Первичный актор:** Администратор группы

**Цель:** Опубликовать важное объявление, которое увидят все участники группы

**Краткое описание основного потока:**
1. Администратор открывает раздел "Объявления" выбранной группы
2. Система отображает форму для создания объявления
3. Администратор вводит заголовок и текст объявления
4. Администратор прикрепляет файл (по желанию)
5. Администратор нажимает кнопку "Опубликовать"
6. Система валидирует данные
7. Система сохраняет объявление в базе данных
8. Система отправляет уведомления участникам (асинхронно)
9. Система отображает сообщение об успехе

**Альтернативные потоки:**
- Планирование публикации на будущее время
- Сохранение объявления как черновика

**Исключительные ситуации:**
- Ошибка валидации данных
- Ошибка сохранения в базу данных
- Таймаут при загрузке файла
- Недоступность сервиса уведомлений

### 5. Анализ границ ответственности

👉 **Ссылка на файл:** [analysis.md](analysis.md)

#### 5.1. Транзакционные границы

| Операция | Синхронная/Асинхронная | Откат при ошибке | Retry-стратегия | Идемпотентность |
|----------|------------------------|------------------|-----------------|-----------------|
| Валидация данных | Синхронная | Да (остановка потока) | Нет | Нет |
| Сохранение объявления в БД | Синхронная | Да (ROLLBACK) | Нет | Да (по ID запроса) |
| Загрузка файла в хранилище | Синхронная | Да (отмена операции) | 3 попытки с exp. backoff | Да (UUID файла) |
| Отправка уведомлений | Асинхронная | Нет (best-effort) | 5 попыток → DLQ | Да (announcement_id + user_id) |

#### 5.2. Обработка исключительных ситуаций

**Реализовано стратегий обработки:** 4

**Примеры:**

##### Исключительная ситуация 1: Таймаут при загрузке файла

- **Условие возникновения:** Внешнее хранилище (S3) не отвечает в течение 5 секунд
- **Обнаружение:** HTTP-клиент выбрасывает TimeoutException, система логирует ошибку
- **Реакция:** Система не сохраняет объявление в БД, прерывает транзакцию, возвращает HTTP 503
- **Компенсация:** Транзакция БД откатывается, фоновый cleanup удаляет частично загруженные файлы
- **Уведомление пользователя:** "Не удалось загрузить файл. Сервис временно недоступен. Попробуйте позже."

##### Исключительная ситуация 2: Ошибка подключения к очереди сообщений

- **Условие возникновения:** После сохранения объявления в БД RabbitMQ недоступен
- **Обнаружение:** AmqpException при вызове rabbitTemplate.convertAndSend()
- **Реакция:** Система логирует ошибку, сохраняет задачу в таблицу outbox со статусом PENDING
- **Компенсация:** Фоновый scheduler сканирует outbox и повторяет отправку каждую минуту
- **Уведомление пользователя:** Пользователь видит "Объявление опубликовано", администратор получает алерт

##### Исключительная ситуация 3: Ошибка валидации данных

- **Условие возникновения:** Пользователь отправил форму с пустым заголовком или превысил лимит файлов
- **Обнаружение:** Сервисный слой выполняет валидацию, нарушены бизнес-правила
- **Реакция:** Система не начинает транзакцию, возвращает HTTP 400 с описанием ошибок
- **Компенсация:** Не требуется, так как никаких изменений не произошло
- **Уведомление пользователя:** UI подсвечивает поля с ошибками, показывает сообщение "Заголовок не может быть пустым"

##### Исключительная ситуация 4: Ошибка сохранения в базу данных

- **Условие возникновения:** Нарушение целостности данных или таймаут подключения к PostgreSQL
- **Обнаружение:** DataIntegrityViolationException или QueryTimeoutException при выполнении SQL
- **Реакция:** Система откатывает транзакцию, логирует ошибку с деталями SQL-запроса
- **Компенсация:** Транзакция автоматически откатывается БД
- **Уведомление пользователя:** "Не удалось опубликовать объявление из-за технической ошибки. Попробуйте позже."

## Контрольные вопросы

**Подготовка к защите:**

1. Что такое транзакционная граница? Где она проходит в вашем сценарии?
   - Транзакционная граница — это область операций, которые должны быть выполнены атомарно (всё или ничего). В моём сценарии транзакционная граница проходит от получения запроса на публикацию до фиксации объявления в базе данных. Сохранение метаданных объявления и информации о файлах происходит в одной транзакции.

2. Почему операция X выбрана синхронной, а Y - асинхронной?
   - Сохранение в БД — синхронная операция, потому что пользователь должен сразу знать, что объявление опубликовано. Отправка уведомлений — асинхронная, так как она не влияет на факт публикации, может выполняться долго (при большом количестве участников), и её временная недоступность не должна блокировать пользователя.

3. Как обеспечить идемпотентность при повторных запросах?
   - Идемпотентность обеспечивается использованием уникального идентификатора запроса (idempotency key), который передаётся клиентом. Сервер проверяет наличие этого ключа в кэше Redis. Если операция с таким ключом уже выполнялась, возвращается ранее сохранённый результат без повторного выполнения.

4. Что произойдёт, если внешний сервис вернёт ошибку после частичного выполнения операции?
   - Если ошибка произошла после сохранения объявления в БД, но до отправки уведомлений (например, RabbitMQ недоступен), задача сохраняется в таблицу outbox. Фоновый процесс позже повторит попытку. Если ошибка произошла во время загрузки файла, транзакция БД откатывается, и объявление не создаётся.

5. Как система обнаружит, что внешний сервис недоступен?
   - Система использует механизмы: таймауты HTTP-клиентов (5-10 секунд), проверку health-эндоинтов сервисов (health checks), обработку исключений (TimeoutException, ConnectException), а также circuit breaker (Hystrix/Resilience4J) для предотвращения каскадных отказов.

6. Какие данные нужно логировать для диагностики сбоев?
   - correlationId (уникальный идентификатор запроса), timestamp (время события), user_id (идентификатор пользователя), operation (название операции), error_message (текст ошибки), stack_trace (трассировка), request_payload (тело запроса для воспроизведения), response_status (код ответа), duration_ms (время выполнения).
