### 3. Интеграционные тесты (БД)

**Назначение**: Проверка взаимодействия PostgreSQL репозитория с реальной БД.

**Технологии**:
- `Testcontainers` — автоматический запуск контейнера PostgreSQL для тестов
- `Alembic` — применение миграций перед тестами
- `pytest` — запуск тестов

**Пример теста (PostgresAnnouncementRepository)**:
```python
def test_save_and_find_by_id(postgres_container):
    repository = PostgresAnnouncementRepository(postgres_container.session)
    
    announcement = Announcement("ann-1", "group-1", "user-1", "Title", "Content")
    repository.save(announcement)
    
    found = repository.find_by_id("ann-1")
    assert found is not None
    assert found.title == "Title"

def test_find_by_group_with_pagination(postgres_container):
    repository = PostgresAnnouncementRepository(postgres_container.session)
    
    # Создаём 10 объявлений
    for i in range(10):
        ann = Announcement(f"ann-{i}", "group-1", "user-1", f"Title {i}", f"Content {i}")
        repository.save(ann)
    
    result = repository.find_by_group("group-1", limit=3, offset=0)
    assert len(result) == 3
    assert result[0].title == "Title 9"  # DESC order
```

**Результаты**:
Протестированы: save, find_by_id, find_by_group, find_by_author, delete

### 4. E2E-тесты

**Назначение**: Проверка полных пользовательских сценариев через реальный HTTP API.

**Технологии**:
- `FastAPI TestClient` — имитация HTTP-запросов к приложению
- Тестовая БД (SQLite in-memory или testcontainers)
- Тестовый сервис уведомлений (консольный)

**Сценарии тестирования**:
Сценарий 1: Успешная публикация объявления
1. POST /api/announcements → создание черновика
2. POST /api/announcements/publish → публикация
3. GET /api/announcements/{id} → проверка статуса "published"

**Пример кода**:
```python
def test_full_publish_workflow(client):
    # Шаг 1: Создание
    create_resp = client.post("/api/announcements/", json={
        "group_id": "group-1",
        "author_id": "user-1",
        "title": "Important",
        "content": "Meeting at 12:00"
    })
    assert create_resp.status_code == 201
    ann_id = create_resp.json()["announcement_id"]
    
    # Шаг 2: Публикация
    publish_resp = client.post("/api/announcements/publish", json={
        "announcement_id": ann_id
    })
    assert publish_resp.status_code == 200
    
    # Шаг 3: Проверка
    get_resp = client.get(f"/api/announcements/{ann_id}")
    assert get_resp.status_code == 200
    assert get_resp.json()["status"] == "published"
```

**Сценарий 2**: Валидация ошибок
1. POST /api/announcements с пустым заголовком → ошибка 400
2. POST /api/announcements с слишком длинным заголовком → ошибка 400
3. GET /api/announcements/несуществующий-id → ошибка 404

**Результаты**:
Всего E2E-сценариев: 6
Покрыты: успешные сценарии, ошибки валидации, пагинация

### 5. CI/CD настройка (GitHub Actions)

Файл: `.github/workflows/tests.yml`
```yaml
name: Tests

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    
    services:
      postgres:
        image: postgres:15
        env:
          POSTGRES_USER: test_user
          POSTGRES_PASSWORD: test_pass
          POSTGRES_DB: test_db
        ports:
          - 5432:5432
        options: >-
          --health-cmd "pg_isready -U test_user"
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
    
    steps:
      - uses: actions/checkout@v3
      
      - name: Setup Python
        uses: actions/setup-python@v4
        with:
          python-version: '3.11'
      
      - name: Install dependencies
        run: |
          pip install -r requirements.txt
          pip install pytest pytest-cov
      
      - name: Run migrations
        run: alembic upgrade head
        env:
          DATABASE_URL: postgresql://test_user:test_pass@localhost:5432/test_db
      
      - name: Run tests
        run: pytest tests/ -v --cov=src --cov-report=xml
        env:
          DATABASE_URL: postgresql://test_user:test_pass@localhost:5432/test_db
      
      - name: Upload coverage
        uses: codecov/codecov-action@v3
        with:
          file: ./coverage.xml
```
