### 1. Repository (PostgreSQL)

**Реализованные методы интерфейса AnnouncementRepository:**
- `save(announcement)` - сохранение агрегата (создание или обновление)
- `find_by_id(announcement_id)` - поиск по ID
- `find_by_group(group_id, limit, offset)` - список объявлений в группе (с пагинацией)
- `find_by_author(author_id, limit, offset)` - список объявлений автора
- `delete(announcement_id)` - удаление объявления

**Технологии:** SQLAlchemy (ORM), psycopg2 (драйвер PostgreSQL)

**ORM модель:**
```python
class AnnouncementModel(Base):
    __tablename__ = "announcements"
    
    id = Column(String(36), primary_key=True)
    group_id = Column(String(36), nullable=False, index=True)
    author_id = Column(String(36), nullable=False, index=True)
    title = Column(String(200), nullable=False)
    content = Column(String(5000), nullable=False)
    status = Column(Enum("draft", "scheduled", "published", "archived"))
    attachments = Column(JSON, default=[])
    created_at = Column(DateTime, nullable=False)
    published_at = Column(DateTime, nullable=True)
    scheduled_for = Column(DateTime, nullable=True)
    updated_at = Column(DateTime, nullable=False)
```

**Пример реализации метода save**:
```python
def save(self, announcement: Announcement) -> None:
    model = self._session.query(AnnouncementModel).filter(
        AnnouncementModel.id == announcement.id
    ).first()
    
    attachments_json = [a.__dict__ for a in announcement.attachments]
    
    if model:
        model.title = announcement.title
        model.content = announcement.content
        model.status = announcement.status.value
        model.attachments = attachments_json
        model.updated_at = announcement.updated_at
    else:
        model = AnnouncementModel(
            id=announcement.id,
            group_id=announcement.group_id,
            author_id=announcement.author_id,
            title=announcement.title,
            content=announcement.content,
            status=announcement.status.value,
            attachments=attachments_json,
            created_at=announcement.created_at,
            published_at=announcement.published_at,
            scheduled_for=announcement.scheduled_for,
            updated_at=announcement.updated_at
        )
        self._session.add(model)
    
    self._session.commit()
```

### 5. Event Publisher (RabbitMQ)

**Реализация NotificationService с RabbitMQ**:
```python
class RabbitMQNotificationService(NotificationService):
    def __init__(self, host: str = "localhost", queue_name: str = "announcement_notifications"):
        self._host = host
        self._queue_name = queue_name
    
    def send_announcement_published(self, announcement_id: str, group_id: str) -> None:
        message = {
            "event_type": "announcement.published",
            "announcement_id": announcement_id,
            "group_id": group_id,
            "timestamp": datetime.now().isoformat()
        }
        self._publish(message)
    
    def _publish(self, message: dict) -> None:
        self._connect()
        self._channel.basic_publish(
            exchange="",
            routing_key=self._queue_name,
            body=json.dumps(message)
        )
```

## Вывод

В ходе выполнения лабораторной работы реализован инфраструктурный слой для сервиса объявлений.
**Что реализовано**:
- PostgreSQL репозиторий с полной реализацией всех методов интерфейса (save, find_by_id, find_by_group, find_by_author, delete)
- SQLAlchemy ORM модель с маппингом доменных объектов
- Alembic миграции для управления схемой БД
- REST API на FastAPI с 11 эндпоинтами (Pydantic валидация, Swagger документация)
- RabbitMQ адаптер для асинхронной публикации событий
- Docker Compose с тремя сервисами (PostgreSQL, RabbitMQ, FastAPI app)
- Интеграционные тесты с покрытием 20 сценариев
**Проблемы**:
- Преобразование доменных объектов в ORM модели требует ручного маппинга
- Настройка RabbitMQ в Docker потребовала дополнительных healthcheck'ов

**Результат**: Полностью работоспособный сервис, готовый к деплою. Все слои (Domain, Application, Infrastructure) интегрированы и общаются через порты/адаптеры. Чистая архитектура соблюдена — замена БД или очереди не потребует изменений в бизнес-логике.
